# OCR 准确度升级设计

## 目标

将当前“图片/PDF 导入 -> OCR -> 用户确认 -> 合同分析”的链路升级为更稳定的文档识别流程，重点解决合同照片识别不准、阅读顺序混乱、截图噪音干扰和多页材料处理能力不足的问题。

升级后的目标链路是：

`文件上传 -> 图片预处理 -> OCR -> 模型校对 -> 合并文本 -> 用户确认 -> 合同分析`

## 范围

- `txt`：继续直接读取文本
- `docx`：继续复用现有提取逻辑
- `jpg/jpeg/png/webp`：支持一次选择多张，按选择顺序串行处理
- `pdf`：
  - 可提取文本的 PDF：优先直接抽取文本
  - 扫描版 PDF：逐页转图片后进入 OCR 链路

不包含：

- 前端拖拽重排页序
- OCR 后自动开始合同分析
- 切换 OCR 校对模型

## 核心决策

### 1. 多张图片串行处理

多张合同照片按用户选择顺序依次处理，不做并行识别。

### 2. OCR 校对固定走模型校对链路

OCR 校对阶段不跟随上传区的其他业务配置变化，而是固定走系统配置里的审查模型，保证校对质量和行为稳定。

### 3. 先确认，再分析

OCR 和校对完成后，前端只把合并后的文本填入右侧文档区，进入 `ocr_ready` 状态。用户确认或手动修改后，才进入现有 `/api/review` 流程。

## 后端设计

新增统一导入入口：

- `POST /api/ocr/ingest`

按输入类型分流：

1. `txt`
   - 直接读取文本
2. `docx`
   - 复用现有 docx 提取逻辑
3. 图片批量上传
   - 按顺序逐张执行：
     - 图片预处理
     - PaddleOCR 初识别
     - 模型校对
4. `pdf`
   - 先尝试直接抽取文本
   - 如果文本为空或明显不足，则逐页转图片并进入同一套 OCR 流程

## 接口返回

```json
{
  "source_type": "image_batch",
  "used_ocr_model": "Qwen/Qwen3.5-4B",
  "merged_text": "完整合同文本",
  "pages": [
    {
      "page_index": 1,
      "filename": "page-1.jpg",
      "text": "第一页文本",
      "average_confidence": 0.97,
      "warnings": []
    }
  ],
  "warnings": []
}
```

字段约定：

- `source_type`：`txt` / `docx` / `image_batch` / `pdf_text` / `pdf_ocr`
- `used_ocr_model`：返回实际参与 OCR 校对的模型标识
- `merged_text`：前端文档区直接展示的文本
- `pages[]`：保留页级结果，便于提示和排错
- `warnings[]`：整体级警告，例如低置信度页提醒

## 前端交互

- 上传区支持：
  - 单个 `txt`
  - 单个 `docx`
  - 单个 `pdf`
  - 多张图片批量选择
- 处理中显示：
  - “正在识别第 1/4 页”
  - “正在进行文字校对”
- 完成后进入 `ocr_ready`
- 用户确认前可直接修改文档文本
- 用户点击“确认并开始分析”后，再进入合同分析

## 失败策略

- 单页失败时，不整单直接中断
- 若仍有成功页：
  - 返回已成功页文本
  - 明确提示失败页
- 如果模型校对失败：
  - 回退到原始 OCR 文本
  - 通过 `warnings[]` 提示用户重点检查
